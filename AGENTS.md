LocalHub 개발 지침

이 문서는 LocalHub 프로젝트를 구현하는 모든 팀원과 VSCode Copilot이 따라야 하는 최상위 개발 지침이다. 구현 판단이 애매하면 기능을 확대하지 말고 이 문서의 MVP 범위와 RFP 요구사항을 우선한다.

## 1. 프로젝트 목표

- 서비스명: LocalHub
- 선정 권역: 서울
- 개발 기간: Day 1~Day 3, 총 3일
- 핵심 문제: 서울의 공식 관광 정보와 실제 이용자의 후기·질문이 분리되어 있어 필요한 장소 정보와 지역 경험을 한 번에 찾기 어렵다.
- 타깃 사용자: 서울 여행·나들이 정보를 빠르게 찾고, 로그인 없이 지역 경험을 공유하려는 사용자
- MVP 목표: 서울 공공데이터 조회, 익명 게시판, 데이터 기반 챗봇, 검색·조회수를 하나의 배포 가능한 서비스로 완성한다.

## 2. 허용 개발 환경 및 AI 사용 정책

- 프로젝트 개발에는 **VSCode와 GitHub Copilot만 사용한다.**
- 서비스 기능 구현을 위한 외부 AI 연동은 고객이 제공한 **OpenAI API Key**만 허용한다.
- Claude Code, Codex, Cursor, Antigravity 등 다른 AI 코딩 도구를 개발 과정에 사용하거나 권장하지 않는다.
- Copilot에 API 키, `.env` 내용, 토큰, 비밀번호 또는 실제 인증 정보를 입력하지 않는다.
- Copilot이 생성한 코드는 반드시 담당자가 diff를 검토하고 직접 실행·테스트한 뒤 반영한다.
- 한 번의 Copilot 요청은 하나의 작은 작업 단위로 제한한다. 관련 없는 리팩터링이나 신규 기능을 함께 생성하지 않는다.
- Copilot 프롬프트에는 수정할 파일, 입력·출력, 완료 조건, 변경 금지 범위를 구체적으로 적는다.

### 권장 Copilot 요청 형식

```text
목표: [이번에 구현할 한 가지 기능]
대상 파일: [수정 허용 파일]
기술 조건: Vue 3 또는 FastAPI + SQLAlchemy + SQLite
완료 조건: [화면/API/테스트에서 확인할 결과]
변경 금지: 로그인 추가, 기술 스택 변경, 범위 밖 리팩터링, 비밀정보 하드코딩
응답: 먼저 구현 계획을 짧게 제시하고, 최소 변경 코드와 검증 방법을 작성해줘.
```

## 3. 고정 기술 스택

- Frontend: Vue.js 3, SPA, 모바일 대응
- Backend: Python, FastAPI, REST API
- ORM/DB: SQLAlchemy, SQLite
- AI API: OpenAI API
- Frontend 배포: Netlify
- Backend 배포: Render
- 환경변수: `.env`; 저장소에는 키가 없는 `.env.example`만 제공

팀 합의와 RFP 변경 없이 프레임워크, DB 또는 배포 서비스를 교체하지 않는다. 3일 일정에 불필요한 라이브러리는 추가하지 않는다.

## 4. Must Have

다음 항목은 일부라도 빠지면 MVP가 완료된 것으로 보지 않는다.

1. 서울 관광 JSON을 연동하고 SQLite·SQLAlchemy 기반으로 지역 및 커뮤니티 데이터를 저장한다.
2. 회원가입·로그인 없는 익명 게시판을 구현한다.
3. 게시글 목록, 상세, 작성, 수정, 삭제를 모두 제공한다.
4. 게시글 작성 시 수정용 비밀번호를 등록하고 수정·삭제 시 일치 여부를 검증한다.
5. RFP의 교육 목적 요구에 따라 수정용 비밀번호는 암호화하지 않고 평문으로 저장·비교한다. 임의로 회원 인증이나 비밀번호 해싱을 추가하지 않는다.
6. FastAPI `POST /api/chat`에서 서울 관광지·축제·음식점 및 커뮤니티 게시글 질의에 응답한다.
7. 챗봇 답변은 제공된 서울 JSON과 커뮤니티 DB를 근거로 생성한다. 데이터에 없는 장소나 사실을 확정적으로 만들지 않는다.
8. Vue 3 SPA에 게시판 화면과 모바일 대응 플로팅 챗봇을 구현하고 대화 히스토리를 유지한다.
9. `.env`와 `.gitignore`를 적용하고 API 키 등 민감정보를 코드, 로그, 문서, Git 이력에 남기지 않는다.
10. 프론트엔드는 Netlify, 백엔드는 Render에 배포하고 외부에서 두 URL이 정상 동작하도록 한다.
11. DB 파일, 기능 명세서, 데이터 출처·라이선스 목록, 배포 URL 등 지정 산출물을 제출한다.

## 5. 선택 기능 우선순위

Must Have가 동작하고 통합된 뒤 아래 순서대로 구현한다.

### Should Have

1. 게시글 검색
   - 제목·내용 키워드 검색
   - 검색 결과 목록 표시
2. 조회수 표시
   - 상세 조회 시 조회수 증가
   - 목록 및 상세 화면에 조회수 노출
3. AI 코스 카드
   - 동행자·날씨·소요 시간을 조건으로 받는다.
   - 제공 서울 JSON에 존재하는 장소만 최대 3개 추천한다.
   - 응답 JSON 형식을 고정하고 API 실패 시 기본 추천 코스를 표시한다.
   - 핵심 기능 완료 후에만 착수한다.

### Could Have

- 서울 축제 캘린더
- 게시글과 장소 자동 연결
- 정보 정정 요청

Could Have는 Must와 Should가 배포 환경에서 검증된 뒤 시간이 남을 때만 검토한다.

## 6. 이번 MVP에서 하지 않는 것

- 사용자 간 실시간 채팅
- 공동 스케줄 공유
- WebSocket 기반 실시간 알림 또는 접속자 현황
- 외부 날씨 API 연동
- 지도 기반 경로 안내
- 다국어 지원
- 복잡한 소셜 공유 및 OG 메타데이터 최적화

Copilot이 위 기능이나 로그인·관리자 시스템을 제안해도 구현하지 않는다. 향후 아이디어는 코드 대신 별도 백로그에만 기록한다.

## 7. 권장 구조와 구현 원칙

프로젝트에 기존 구조가 있으면 그 구조를 우선한다. 새로 구성할 경우 다음 경계를 권장한다.

```text
frontend/
  src/
    components/
    views/
    services/
    router/
backend/
  app/
    routers/
    schemas/
    models/
    services/
    database.py
    main.py
data/
docs/
```

- Vue 컴포넌트는 화면, 재사용 컴포넌트, API 호출 책임을 분리한다.
- FastAPI 라우터는 요청 검증과 HTTP 응답을 담당하고, 검색·챗봇·데이터 처리 로직은 서비스 계층으로 분리한다.
- Pydantic 스키마로 요청과 응답 형식을 명시한다.
- 게시글 수정·삭제 API는 비밀번호 누락, 불일치, 존재하지 않는 게시글을 구분해 처리한다.
- 검색은 우선 제목·내용의 단순 부분 일치로 구현한다. 3일 MVP에 전문 검색 엔진을 도입하지 않는다.
- AI 응답에 전달할 데이터는 질문과 관련된 최소 범위로 제한한다.
- 사용자 입력, API 오류, 빈 검색 결과, 빈 데이터 상태를 화면에서 명확히 표시한다.
- 기존 동작을 깨는 대규모 리팩터링보다 작고 검증 가능한 변경을 우선한다.

## 8. API 및 데이터 규칙

- REST 경로와 HTTP 메서드는 일관되게 유지한다.
- 날짜·시간과 API 필드명 형식을 프론트엔드와 백엔드가 사전에 합의한다.
- 서울 JSON 원본은 가능한 한 보존하고, 정제·변환 로직을 별도로 둔다.
- 추가 데이터를 수집할 경우 출처, 라이선스, 수집일을 기능 명세서에 기록한다.
- OpenAI API 오류, 타임아웃, 빈 응답을 처리하고 사용자에게 재시도 가능한 메시지를 제공한다.
- OpenAI 호출량과 프롬프트 크기를 제한하여 제공 키의 예산을 초과하지 않는다.

## 9. 3일 실행 계획

### Day 1 — 기획·설계·기반 구축

- 서울 권역과 제공 JSON 확정
- RFP 요구사항 및 API 계약 확정
- 사용자 흐름과 Vue 화면 구조 설계
- SQLite 스키마와 SQLAlchemy 모델 설계
- 초기 데이터 적재

### Day 2 — 핵심 개발

- 게시글 CRUD와 수정용 비밀번호 검증
- 게시글 검색과 조회수
- 커뮤니티 Vue 화면
- 플로팅 챗봇 UI
- `POST /api/chat` 및 데이터 기반 답변 처리

### Day 3 — 통합·배포·제출

- Netlify·Render 배포 및 CORS·환경변수 검증
- CRUD·검색·챗봇 통합 테스트
- DB, 기능 명세서, 라이선스 목록, 배포 URL 정리
- 발표자료와 시연 시나리오 준비

일정이 지연되면 AI 코스 카드와 Could Have를 즉시 중단하고 Must Have의 통합·배포를 우선한다.

## 10. 리스크 대응

- JSON 결측치·형식 불일치: Day 1에 필드별 점검 후 필요한 항목만 안전하게 정규화한다. 없는 값은 빈 문자열로 날조하지 말고 `null` 또는 명시적 대체 문구로 처리한다.
- OpenAI API 실패·환각: 제공 데이터만 사용하도록 프롬프트를 제한하고, 고정 응답 형식과 기본 대체 응답을 준비한다.
- Render·SQLite 영속성: Day 1~2에 재시작·재배포 후 데이터 유지 여부를 확인하고 초기 데이터 및 복구 절차를 준비한다.
- 배포 일정 지연: Day 3 오전까지 핵심 CRUD와 챗봇을 우선 배포하며 선택 기능은 완료 기준을 충족한 뒤 착수한다.
- 비밀정보 노출: 커밋 전 `.env`, API 키 패턴, 로그를 점검하고 노출 시 즉시 키를 폐기·재발급한다.

## 11. 완료 기준

다음을 모두 만족해야 작업을 완료로 표시한다.

- 게시글 CRUD와 비밀번호 기반 수정·삭제가 정상 동작한다.
- 제목·내용 키워드 2개 이상으로 검색 결과가 정확히 표시된다.
- 관광지·축제·음식점 대표 질문 5개 중 4개 이상을 제공 데이터에 근거해 답변한다.
- 모바일과 데스크톱에서 게시판 및 플로팅 챗봇을 사용할 수 있다.
- Netlify와 Render 외부 URL이 정상 접속되고 프론트엔드에서 API를 호출할 수 있다.
- `.env`와 실제 키가 저장소 및 제출 산출물에 포함되지 않는다.
- SQLite DB와 초기 데이터, 기능 명세서, 데이터 출처·라이선스, 배포 URL을 제출한다.
- 담당자가 Copilot 생성 코드를 검토하고 핵심 기능을 직접 테스트했다.

## 12. 변경 통제

- 개발 착수 후 새로운 기능을 Must Have로 추가하지 않는다.
- 요구사항 변경이 필요하면 변경 이유, 일정 영향, 포기할 기능을 먼저 기록하고 팀 전원 합의 후 반영한다.
- 범위 충돌 시 우선순위는 `RFP Must Have → 배포 안정성 → 선택된 검색·조회수 → AI 코스 카드 → Could Have` 순이다.

## 13. 기능 개발 워크플로우 (필수)

- 백엔드 기능 개발은 API 1개 단위로 진행한다.
- 각 API 작업은 아래 순서를 반드시 따른다.
   1) 구현
   2) 실행 검증(성공/오류 케이스)
   3) 문서 반영
   4) Git 커밋
- 커밋 메시지는 기능 중심 한국어로 작성하고, API 상세 경로 문자열(`/api/...`)은 메시지에 쓰지 않는다.
- 기능 구현 시 아래 문서를 같은 작업 단위로 함께 갱신한다.
   - `docs/backend/development-log.md`
   - `docs/backend/troubleshooting.md`
   - `docs/backend/api.md`
- 오류가 발생하면 원인, 조치, 결과를 `docs/backend/troubleshooting.md`에 즉시 기록한다.

## 14. 데이터 파일 및 스키마 사용 규칙 (필수)

- 서울 데이터 적재는 `data/SCHEMA.md`에 정의된 TourAPI 4.0 원본 형식을 기준으로 구현한다.
- 기본 입력 데이터는 `data/서울_*.json` 파일군을 사용한다.
- 원본 구조(`region`, `contentType`, `contentTypeId`, `items[]`)를 임의 변형하지 않는다.
- 데이터 출처·라이선스 표기는 `data/SOURCE.md`를 기준으로 유지한다.
- 적재 로직 변경 시 스키마 불일치 처리(결측/타입/빈값) 규칙을 문서에 남긴다.
